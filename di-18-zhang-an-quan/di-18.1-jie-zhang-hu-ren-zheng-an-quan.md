# 18.1 账户认证安全

FreeBSD 通过 /etc/master.passwd 主密码文件与 pam_unix.so 认证模块实现账户安全管理。本节涵盖密码数据库生成流程、账户锁定与 nologin 禁止登录、SHA512 哈希验证及 pam_passwdqc 密码策略配置。

## 密码文件目录结构

在 FreeBSD 中 `/etc/passwd` 并非实际使用的密码文件，仅出于兼容目的保留，真实使用的用户密码文件是 `/etc/master.passwd`。

```sh
/etc/master.passwd
        │
        │  pwd_mkdb(8)
        ▼
────────────────────────────
      数据库生成过程
────────────────────────────
        │
        ├──────────────┬──────────────┐
        ▼              ▼              ▼
/etc/pwd.db     /etc/spwd.db     /etc/passwd
（db(3)格式）    （db(3)格式）     （ASCII 密码兼容文件）
（无密码）        （含密码）        （无密码）
```

## 阻止登录

保护系统的第一步，通常是对账户进行审计。禁用所有无需登录访问的账户。

>**思考题**
>
>>确保 `root` 拥有一个强密码，并应定期轮替密码，且不应共享该密码。
>
>这种说法很流行，但这一做法背后存在安全隐患。
>
> - 强密码的复杂性使得其难以记忆，进而需要依赖不安全的存储方式，如纸质记录或文本文件。这种存储方式可能带来潜在的安全风险。
> - 定期更换密码进一步增加了密码管理的复杂性，而“密码管理工具”本身亦面临被信任的问题。
> - 尽管基于密钥的认证方式提供了一定的安全性，但其密钥丢失的风险使得身份验证过程更加复杂。
> - 此外，多因素认证（MFA）虽能增强安全性，但其依赖的设备一旦丢失或程序卸载，将造成类似的问题。

拒绝账户登录访问的手段有两种。

第一种是锁定账户。以下示例展示如何锁定账户 `ykla`：

```sh
# pw lock ykla
```

锁定用户 `ykla` 尝试登录时：

```sh
FreeBSD/amd64 (ykla) (ttyv1)

login: ykla
Password:
Login incorrect
login:
```

其他用户尝试切换到用户 `ykla` 时：

```sh
# su ykla
su: Sorry
```

>**警告**
>
>请勿尝试锁定 root 账户。
>
>```sh
>ykla@ykla:~ $ su
>Password:
>su: Sorry
>```
>
>否则将可能需要进入单用户模式进行恢复。

要解除用户 `ykla` 的锁定，执行：

```sh
# pw unlock ykla
```

第二种方法是将 Shell 改为 `/usr/sbin/nologin`，从而阻止登录访问。当用户尝试登录时，nologin  Shell 会阻止系统为其分配 Shell。

只有 root 有权更改其他用户的 Shell：

```sh
# chsh -s /usr/sbin/nologin ykla
```

用户 `ykla` 登录时被阻拦的效果与前例类似。

要想恢复登录，重新为用户 `ykla` 分配 sh：

```sh
# chsh -s sh ykla
```

>**警告**
>
>请勿尝试为 root 账户分配 `/usr/sbin/nologin`。

## 密码哈希

密码是技术领域必不可少的负担。既然必须使用，密码应当足够复杂，其存储在密码数据库中的版本，应当借助强大的哈希机制加密。FreeBSD 的 `crypt()` 库支持包括 SHA256、SHA512 和 Blowfish 等多种算法。

不应将默认的 SHA512 降级为更弱的哈希算法，也不应替换为 Blowfish。

Blowfish 不属于 AES 家族，也不符合联邦信息处理标准（FIPS），在某些环境中可能无法通过合规性审查。并且 Blowfish 算法本身因分组长度过短已被认为不安全。

### 查看当前用户密码使用的哈希

若要确认某位用户密码所使用的哈希算法，root 用户可以查看 FreeBSD 密码数据库中对应哈希值。每个哈希值均以一个符号开头，表明用于加密该密码的哈希机制类型：

- 无起始符号：DES
- `$`：MD5
- `$6$`：SHA256 和 SHA512
- `$2a$`：Blowfish

本例中，`ykla` 的密码以 `$6$` 开头，说明其使用了默认的 SHA512 算法。

```sh
# grep ykla /etc/master.passwd
```

注意，密码数据库中存放的是加密后的哈希值，而非原始密码。输出应类似于以下内容：

```sh
ykla:$6$SqMJXrv5aC6Wq.by$nmbZs078aHNBVyh9noLFouJsGHyFSvQIzH0W4zpdfXuPtGtt.FHgWfXDHVBa.g9P0eZ32UwfByzRKdVnTaO7W.:1001:1001::0:0:User &:/home/ykla:/bin/sh
```

### 在登录类别中设置默认密码哈希

还可以在用户的登录类别中设置哈希机制。

可以运行以下命令来检查当前正在使用的哈希机制：

```sh
# grep passwd_format /etc/login.conf
```

输出应类似于以下内容：

```ini
	:passwd_format=sha512:\
#	:passwd_format=des:\ # 这是注释行，可忽略
```

例如，要切换为 Blowfish，可将该行修改为：

```ini
:passwd_format=blf:\
```

接着，必须执行 `cap_mkdb` 来更新 login.conf 数据库：

```sh
# cap_mkdb /etc/login.conf
```

注意，此更改不会影响已有的密码哈希。也就是说，所有用户都需要运行 `passwd` 修改一次密码，才能让新算法生效。

### 密码策略执行

为本地账户实施强密码策略，是系统安全的一项基本配置。FreeBSD 可借助内置的可插拔认证模块（PAM）控制密码长度、强度与复杂度。

本节演示如何利用 `pam_passwdqc` 模块设置密码的最短与最长长度，并强制要求混合字符。该模块在用户修改密码时生效。

>**警告**
>
>强制要求密码混合字符，密码轮替、禁用常见词汇等复杂性规则，已不再被视为最佳安全实践。
>
>分析这两种安全哲学。

配置前，先切换为 root 用户，再将 `/etc/pam.d/passwd` 文件中包含 `pam_passwdqc.so` 的那一行取消注释。

```sh
#
#
# PAM configuration for the "passwd" service
#

# passwd(1) does not use the auth, account or session services.

# password
#password       requisite       pam_passwdqc.so         enforce=users # 注意此行，移除行首的 # 符号
password        required        pam_unix.so             no_warn try_first_pass nullok
```

然后，按密码策略编辑该行，实现禁用一、二、三类密码（包括密码短语），仅允许四类字符密码（≥10位）或三类字符密码（≥12位），新旧密码必须不相似，失败最多重试3次，仅对普通用户强制执行强密码规则（root不强制）。

```ini
password        requisite       pam_passwdqc.so         min=disabled,disabled,disabled,12,10 similar=deny retry=3 enforce=users
```

参数含义：

`min=disabled,disabled,disabled,12,10` 对应五种密码复杂度：

1. 仅一种字符（disabled, 禁用）
2. 仅两种字符（disabled, 禁用）
3. **密码短语**（disabled, 禁用）
4. 包含三种字符（启用但是最小 12 位）
5. 包含四种字符（启用但是最小 10 位）

保存文件后，用户修改密码时会看到类似下面的提示：

```sh
$ passwd
```

输出应类似于以下内容：

```text
Changing local password for ykla
Old Password:

You can now choose the new password.
A valid password should be a mix of upper and lower case letters,
digits and other characters.  You can use a 12 character long
password with characters from at least 3 of these 4 classes, or
a 10 character long password containing characters from all the
classes.  Characters that form a common pattern are discarded by
the check.
Alternatively, if noone else can see your terminal now, you can
pick this as your password: "embark,France_list".
```

若输入的密码不符合策略，系统会拒绝并给出警告。如：

```sh
Weak password: too short.
```

用户可在配置的重试次数内反复尝试。

若组织要求密码定期过期，FreeBSD 可在 /etc/login.conf 的用户登录类别中设置 `passwordtime` 参数。

`default` 登录类别中包含一个示例：

```ini
#       :passwordtime=90d:\
```

因此，若想为此登录类别设定 90 天过期，只需移除注释符号（`#`），保存文件并执行：

```sh
# cap_mkdb /etc/login.conf
```

若要为单个用户 `ykla` 设置过期时间，可向 `pw` 同时传递过期日期（或距离过期的天数）和用户名：

```sh
# pw usermod -p 31-05-2026 -n ykla
```

过期日期以日、月、年的格式设置（`dd-mm-yy[yy]`）。

禁用过期：如果需要取消过期限制，通常将其设为 0。

```sh
# pw usermod -p 0 -n ykla
```

也可以指定相对日期。从今天起，指定用户 ykla 在 30 天后过期：

```sh
# pw usermod ykla -e +30d
```

从今天起，指定用户 ykla 在一年后过期：

```sh
# pw usermod ykla -e +1y
```
