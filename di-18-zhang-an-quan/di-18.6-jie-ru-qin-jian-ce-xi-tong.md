# 18.6 入侵检测系统（IDS）

mtree 通过种子值生成目录的校验和规范，并可事后比对识别文件变更。本节演示规范创建、模拟篡改与校验验证的完整流程。

## 入侵检测系统（IDS）

验证系统文件和二进制文件的完整性，是系统管理与安全团队掌握变更情况的重要手段。凡能监控系统变更的软件，统称为入侵检测系统（Intrusion Detection System, IDS）。

FreeBSD 原生内置了一个叫作 mtree 的基本 IDS。虽然每夜安全邮件会通报变更，但信息毕竟存储在本地，恶意用户仍有机会篡改这些记录以掩盖其操作。因此，应当另建一套独立的二进制签名，保存在只读、属主为 root 的目录中；当然，放在可移动 USB 磁盘或远程服务器上更为理想。

此外，建议在每次更新后运行 `freebsd-update IDS`。

## 生成规范文件

内置的 mtree 工具可生成目录内容的规范。规范依赖一个种子值（一个数值常量）来生成，后续检查规范是否改变也同样需要它，由此即可判断文件或二进制文件是否被修改过。因为攻击者无从知晓种子值，伪造或比对校验和将极其困难，甚至不可能做到。

建议为存放二进制文件和配置文件的目录，以及所有包含敏感数据的目录生成规范。常见的对象包括 **/bin**、**/sbin**、**/usr/bin**、**/usr/sbin**、**/usr/local/bin**、**/etc** 和 **/usr/local/etc**。

下面的示例为 **/bin** 下的每个系统二进制文件生成一组 `sha512` 哈希值，并存入用户主目录下的隐藏文件 **/home/user/.bin_chksum_mtree**：

```sh
# mtree -s 123456789 -c -K cksum,sha512 -p /bin > /home/user/.bin_chksum_mtree
```

输出应类似于以下内容：

```text
mtree: /bin checksum: 3427012225
```

 `123456789` 代表种子值，应随机选取。种子值应牢记，且不应泄露。

同时也要对恶意用户隐藏种子值与校验和输出结果。

mtree 格式是一种描述文件系统对象集合的文本格式，通常用于创建或验证目录层次结构。

mtree 文件由一系列行组成，每行描述一个文件系统对象（前导空白始终被忽略）。

## 验证规范文件

验证二进制签名是否发生变更，只需将目录当前内容与先前生成的规范比较，结果存入文件即可。

该命令需要当初生成原始规范时所用的那个种子值：

```sh
# mtree -s 123456789 -p /bin < /home/user/.bin_chksum_mtree >> /home/user/.bin_chksum_output
```

该命令应当输出与创建规范时一致的 **/bin** 校验和。若目录中的二进制文件未曾变更，**/home/user/.bin_chksum_output** 将是一个空文件。

为了模拟一次变更，可先用 `touch` 修改 **/bin/cat** 的时间戳，再执行验证命令：

```sh
# touch /bin/cat
```

再次运行验证命令：

```sh
# mtree -s 123456789 -p /bin < /home/user/.bin_chksum_mtree >> /home/user/.bin_chksum_output
```

然后检查输出内容：

```sh
# cat /root/.bin_chksum_output
```

输出应类似于以下内容：

```text
cat:    modification time (Fri Aug 25 13:30:17 2023, Fri Aug 25 13:34:20 2023)
```

这只是命令输出示例，旨在说明元数据被修改后会出现的情况。

## 课后习题

1. 使用 `mtree` 为 **/etc** 目录创建完整性基线，然后人为修改 **/etc/motd** 并添加一个用户，再次运行 `mtree` 检测变更。将检测输出与你的修改操作逐一对应，说明每条检测记录的含义。

2. 查阅 AIDE（Advanced Intrusion Detection Environment）的文档，比较 `mtree` 与 AIDE 在功能特性上的差异，说明在生产环境中为何通常选择 AIDE 而非系统自带的 `mtree`。

3. 设计一个实验：安装一个 rootkit 演示工具（如 `chkrootkit`），运行后阅读其检测逻辑，结合 IDS 的原理分析为什么完整性校验工具能够或不能检测出内核级 rootkit。
