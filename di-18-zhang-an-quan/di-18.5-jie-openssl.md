# 18.5 OpenSSL

OpenSSL 3 自 FreeBSD 14 起纳入基本系统，引入提供者模块（provider module）架构。本节涵盖自签证书生成（自签 CA）、FIPS 合规配置（fips_module 完整性检查与 openssl-fipsinstall）及已弃用算法加载。

## 概述

OpenSSL 是一款加密工具包，实现了安全套接字层（SSL）与传输层安全（TLS）网络协议，同时提供丰富的加密例程。

`openssl` 程序是该工具包的命令行入口，让用户可在 Shell 中调用 OpenSSL 加密库的各类功能，包括：

- 私钥、公钥和参数的创建与管理
- 公钥加密操作
- X.509 证书、CSR 和 CRL 的创建
- 消息摘要的计算
- 使用密码进行加密和解密
- SSL/TLS 客户端和服务器测试
- S/MIME 签名或加密邮件的处理
- 时间戳请求、生成和验证
- 加密例程的性能测试

更多信息请参阅免费的 [OpenSSL Cookbook](https://www.feistyduck.com/books/openssl-cookbook/)。

## 生成证书

OpenSSL 支持生成证书，既可供 CA 验证，也可自行使用。

执行 openssl 并搭配以下参数，即可生成供 CA 签署的证书。该命令会在当前目录创建两个文件：证书请求文件 req.pem 可递交至 CA，由 CA 验证凭据、签署后返回已签名的证书；另一个文件 cert.key 则是证书的私钥，务必妥善保管。私钥一旦落入他人之手，即可被用于冒充用户或服务器。

执行以下命令生成证书：

```sh
# openssl req -new -nodes -out req.pem -keyout cert.key -sha3-512 -newkey rsa:4096
```

解释命令选项：

| 选项 | 解释 |
| ---- | ---- |
| `openssl req` | 生成证书签名请求（CSR，Certificate Signing Request） |
| `-new` | 创建一个新的证书签名请求 |
| `-nodes` | 不为私钥设置密码（no DES），生成的私钥不会加密保护 |
| `-out req.pem` | 指定生成的证书签名请求文件的输出文件名，这里是 `req.pem` |
| `-keyout cert.key` | 指定生成的私钥文件的输出路径，这里是 `cert.key` |
| `-sha3-512` | 使用 SHA-3 512 位哈希算法进行签名请求的哈希计算 |
| `-newkey rsa:4096` | 生成新的 RSA 密钥对，使用 4096 位的 RSA 加密算法。 |

若无需 CA 签署，也像上面这样可创建自签名证书。将生成两个新文件：私钥文件 cert.key 和证书文件 cert.crt。

```sh
# ls
req.pem		cert.key
```

建议将上述两个文件存放于 **/etc/ssl/** 下，权限设为 `0700`（仅 `root` 可读），可用 `chmod` 设置。

```sh
# cp req.pem cert.key /etc/ssl/
# chmod 0700 /etc/ssl/req.pem 
# chmod 0700 /etc/ssl/cert.key
```

执行以下命令生成证书：

```sh
# openssl req -new -x509 -days 365 -sha3-512 -keyout /etc/ssl/cert.key -out /etc/ssl/certs/cert.crt
```

## 配置 FIPS 提供者

OpenSSL 3 自 FreeBSD 14 起被纳入基本系统，随之而来的还有全新的提供者模块（provider module）概念。

除内置于库中的默认提供者外，`legacy` 模块负责实现如今已变为可选的、已弃用的加密算法，而 `fips` 模块则严格将 OpenSSL 实现限定在 FIPS 标准所允许的密码算法集合内。

OpenSSL 的 FIPS 相关部分得到了[重点关注](https://www.openssl.org/docs/fips.html)，附有[已知安全问题列表](https://www.openssl.org/news/fips-cve.html)，并定期接受 [FIPS 140 验证流程](https://github.com/openssl/openssl/blob/master/README-FIPS.md)的审查。[FIPS 已验证版本清单](https://www.openssl.org/source/)也已公开。这些措施让用户在使用 OpenSSL 时得以确保满足 FIPS 合规要求。

fips_module 还受到额外的安全机制保护，未通过完整性检查则无法使用。本地系统管理员可按需配置此检查，继而允许每个 OpenSSL 3 用户加载该模块。若配置不当，FIPS 模块会按如下方式报错：

```sh
# echo test | openssl aes-128-cbc -a -provider fips -pbkdf2
```

输出应类似于以下内容：

```text
aes-128-cbc: unable to load provider fips
Hint: use -provider-path option or OPENSSL_MODULES environment variable.
1090A56510220000:error:12800067:DSO support routines:dlfcn_load:could not load the sharedlibrary:/usr/src/crypto/openssl/crypto/dso/dso_dlfcn.c:115:filename(/usr/lib/ossl-modules/fips.so): Cannot open "/usr/lib/ossl-modules/fips.so"
1090A56510220000:error:12800067:DSO support routines:DSO_load:could not load the shared library:/usr/src/crypto/openssl/crypto/dso/dso_lib.c:147:
1090A56510220000:error:07880025:common libcrypto routines:provider_init:reason(37):/usr/src/crypto/openssl/crypto/provider_core.c:1026:name=fips
```

可通过创建 **/etc/ssl/fipsmodule.cnf** 来配置完整性检查，并在 OpenSSL 主配置文件 **/etc/ssl/openssl.cnf** 中引用。OpenSSL 提供了 `openssl-fipsinstall` 辅助此过程，用法如下：

```sh
# openssl fipsinstall -module /usr/lib/ossl-modules/fips.so -out /etc/ssl/fipsmodule.cnf
```

输出应类似于以下内容：

```text
INSTALL PASSED
```

接下来应修改 **/etc/ssl/openssl.cnf**，使其完成以下三项：

- 引入上面生成的 **/etc/ssl/fipsmodule.cnf** 文件，
- 导出 FIPS 模块以供调用，
- 并显式激活默认模块。

```ini
[...]
# For FIPS
# Optionally include a file that is generated by the OpenSSL fipsinstall
# application. This file contains configuration data required by the OpenSSL
# fips provider. It contains a named section e.g. [fips_sect] which is
# referenced from the [provider_sect] below.
# Refer to the OpenSSL security policy for more information.
.include /etc/ssl/fipsmodule.cnf

[...]

# List of providers to load
[provider_sect]
default = default_sect
# The fips section name should match the section name inside the
# included fipsmodule.cnf.
fips = fips_sect

# If no providers are activated explicitly, the default one is activated implicitly.
# See man 7 OSSL_PROVIDER-default for more details.
#
# If you add a section explicitly activating any other provider(s), you most
# probably need to explicitly activate the default provider, otherwise it
# becomes unavailable in openssl.  As a consequence applications depending on
# OpenSSL may not work correctly which could lead to significant system
# problems including inability to remotely access the system.
[default_sect]
activate = 1
```

以上步骤完成后，即可确认 FIPS 模块确实可用且正常工作：

```sh
# echo test | openssl aes-128-cbc -a -provider fips -pbkdf2
```

输出应类似于以下内容：

```text
enter AES-128-CBC encryption password:
Verifying - enter AES-128-CBC encryption password:
U2FsdGVkX18idooW6e3LqWeeiKP76kufcOUClh57j8U=
```

每当 FIPS 模块发生变动——例如系统更新后，或应用了影响基本系统中 OpenSSL 的安全修复之后——都必须重复执行上述过程。

## 课后习题

1. 使用 `openssl speed` 命令在本地 FreeBSD 系统上分别测试 AES-128-CBC、AES-256-GCM 和 ChaCha20-Poly1305 三种加密算法的吞吐量，分析结果差异的原因。

2. 用 `openssl s_client -connect` 连接一个公网 HTTPS 站点，从输出中提取证书链、协议版本和协商的密码套件，绘制证书的信任链结构图。

3. 使用 `openssl req -x509` 生成一个自签名证书，然后配置 FreeBSD 上的 Nginx 或 Apache 使用该证书提供 HTTPS 服务。
