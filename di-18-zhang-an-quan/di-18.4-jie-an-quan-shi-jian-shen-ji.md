# 18.4 安全事件审计

进程记账记录每条命令的执行信息，可用于追踪入侵途径。本节演示使用 `accton` 启用记账、`lastcomm` 查看命令历史及 `sa` 统计资源用量。

## 进程记账

进程记账是一种安全手段，管理员可借此追踪系统资源的使用情况及其在不同用户间的分配，提供系统监控功能，并在最低限度上追踪用户执行的命令。

进程记账有利有弊。其优势在于可以将入侵追溯到入侵点；而弊端是它产生的日志量庞大，对磁盘空间有较高要求。

### 启用和使用进程记账

在使用进程记账之前，必须使用以下命令先行启用：

```sh
# service accounting enable
# service accounting start
```

记账信息存储在 **/var/account** 目录下，该目录会在记账服务首次启动时按需自动创建。这些文件包含敏感信息，涵盖所有用户执行过的每一条命令。文件的写入权限仅限于 `root`，读取权限仅限于 `root` 和 `wheel` 组成员。若要同时阻止 `wheel` 成员读取，可将 **/var/account** 目录的权限模式改为仅允许 `root` 访问。

启用后，记账将开始跟踪 CPU 统计信息、所执行命令等数据。所有记账日志均以非人类可读的格式存储，须通过 `sa` 查看。若不加任何选项执行，`sa` 会打印每位用户的调用次数、总经过时间（分钟）、总 CPU 和用户时间（分钟）以及平均 I/O 操作次数等信息。

要显示用户执行的命令，使用 `lastcomm`。

例如，此命令将打印 `trhodes` 在 `ttyp1` 终端上所有 `ls` 的使用记录：

```sh
# lastcomm ls trhodes ttyp1
```

## 安全事件审计概述

FreeBSD 操作系统包含对安全事件审计的支持。事件审计支持对各类安全相关系统事件进行可靠、精细且可配置的日志记录，包括登录、配置变更、文件与网络访问。这些日志记录对于实时系统监控、入侵检测和事后分析具有不可估量的价值。FreeBSD 实现了 Sun™ 发布的 Basic Security Module（BSM）应用程序编程接口（API）及文件格式，可与 Solaris™ 和 macOS 的审计实现互操作。

本章重点介绍事件审计的安装与配置，阐释审计策略，并提供一个审计配置示例。

审计设施存在一些已知局限性。并非所有安全相关系统事件均可审计，且部分登录机制（如基于 Xorg 的显示管理器和第三方守护进程）无法正确为用户的登录会话配置审计。

安全事件审计设施能够生成非常详细的系统活动日志。在繁忙的系统上，若配置为高细节级别，跟踪文件数据量可能非常庞大，某些配置下每周可超过数 GB。管理员应考虑高数据量审计配置所带来的磁盘空间需求。例如，可能需要为 **/var/audit** 分配一个专用文件系统，以便在审计文件系统写满时不致影响其他文件系统。

### 安全事件审计关键术语

以下术语与安全事件审计相关：

- *事件（event）*：可审计事件是指任何可使用审计子系统记录的事件。安全相关事件的示例包括创建文件、建立网络连接或某用户登录。事件分为“有归属”（attributable）——即可以追踪到已认证用户——和“无归属”（non-attributable）两类。无归属事件的例子包括登录过程中认证之前发生的任何事件，例如错误的密码尝试。
- *类（class）*：一组命名相关事件，用于选择表达式。常用事件类包括“文件创建”（fc）、“执行”（ex）和“登录_注销”（lo）。
- *记录（record）*：描述安全事件的审计日志条目。记录包含记录事件类型、执行操作的主体（用户）信息、日期和时间信息、涉及对象或参数的信息，以及成功或失败条件。
- *跟踪（trail）*：由一系列描述安全事件的审计记录组成的日志文件。跟踪按事件完成时间大致按时间顺序排列。仅授权进程允许将记录提交给审计跟踪。
- *选择表达式（selection expression）*：一个字符串，包含前缀列表和审计事件类名称，用于匹配事件。
- *预选择（preselection）*：系统确定管理员关注哪些事件的过程。预选择配置使用一系列选择表达式，确定对哪些用户审计哪些事件类，以及同时适用于已认证和未认证进程的全局设置。
- *精简（reduction）*：从现有审计跟踪中筛选记录以供保存、打印或分析的过程；同样也是从审计跟踪中移除不需要的审计记录的过程。管理员可借助精简实施审计数据保存策略。例如，详细审计跟踪可保留一个月，之后可精简跟踪，仅保留登录信息以供归档。

### 审计配置

事件审计的用户空间支持已作为 FreeBSD 基础操作系统的一部分安装。GENERIC 内核默认提供内核支持，auditd 可通过在 /etc/rc.conf 中添加以下行来启用：

```
auditd_enable="YES"
```

然后，启动审计守护进程：

```
# service auditd start
```

偏好编译自定义内核的用户必须在其自定义内核配置文件中加入以下行：

```
options	AUDIT
```

#### 事件选择表达式

选择表达式在审计配置的多处位置用于确定应审计哪些事件。表达式包含要匹配的事件类列表。选择表达式从左至右求值，将两个表达式合并即为将一个追加到另一个之后。

以下汇总了默认审计事件类：

默认审计事件类

| 类名 | 描述 | 操作 |
| ---------- | ----------- | ------ |
| all | 全部 | 匹配所有事件类。 |
| aa | 认证与授权 | |
| ad | 管理 | 在系统整体层面执行的管理操作。 |
| ap | 应用程序 | 应用程序定义的操作。 |
| cl | 文件关闭 | 审计对 `close` 系统调用的调用。 |
| ex | 执行 | 审计程序执行。命令行参数和环境变量的审计通过 audit_control 中的 `policy` 设置，使用 `argv` 和 `envv` 参数控制。 |
| fa | 文件属性访问 | 审计对象属性的访问，如 stat(1) 和 pathconf(2)。 |
| fc | 文件创建 | 审计文件创建相关事件。 |
| fd | 文件删除 | 审计文件删除相关事件。 |
| fm | 文件属性修改 | 审计文件属性修改相关事件，如 chown(8)、chflags(1) 和 flock(2)。 |
| fr | 文件读取 | 审计读取数据或为读取而打开文件的事件。 |
| fw | 文件写入 | 审计写入数据或写入/修改文件的事件。 |
| io | ioctl | 审计 `ioctl` 系统调用的使用。 |
| ip | 进程间通信 | 审计各种形式的进程间通信，包括 POSIX 管道和 System V IPC 操作。 |
| lo | 登录_注销 | 审计 login(1) 和 logout(1) 事件。 |
| na | 无归属 | 审计无归属事件。 |
| no | 无效类 | 不匹配任何审计事件。 |
| nt | 网络 | 审计与网络操作相关的事件，如 connect(2) 和 accept(2)。 |
| ot | 其他 | 审计其他杂项事件。 |
| pc | 进程 | 审计进程操作，如 exec(3) 和 exit(3)。 |

这些审计事件类可通过修改 audit_class 和 audit_event 配置文件来进行自定义。

每个审计事件类可附带一个前缀，指示是否匹配成功/失败的操作，以及该条目是为该类及类型增加还是移除匹配。审计事件类前缀汇总如下：

审计事件类前缀

| 前缀 | 操作 |
| ------ | ------ |
| + | 审计该类中成功的事件。 |
| - | 审计该类中失败的事件。 |
| ^ | 既不审计该类中成功也不审计失败的事件。 |
| ^+ | 不审计该类中成功的事件。 |
| ^- | 不审计该类中失败的事件。 |

若无前缀，则该事件的成功和失败实例均将被审计。

以下示例选择字符串选择成功和失败的登录/注销事件，但仅选择成功的执行事件：

```
lo,+ex
```

#### 配置文件

安全事件审计的以下配置文件位于 **/etc/security** 中：

- audit_class：包含审计类定义。
- audit_control：控制审计子系统的各方面，如默认审计类、审计日志卷上保留的最小磁盘空间以及最大审计跟踪大小。
- audit_event：系统审计事件的文本名称与描述，以及每个事件所属的类列表。
- audit_user：用户特定的审计要求，在登录时与全局默认值合并。
- audit_warn：一个由 auditd 使用的可自定义 shell 脚本，用于在异常情况下生成警告消息，例如审计记录空间即将耗尽或审计跟踪文件已完成轮转时。

审计配置文件应谨慎编辑和维护，因为配置错误可能导致事件日志记录不当。

在大多数情况下，管理员仅需修改 audit_control 和 audit_user。前者控制系统范围的审计属性和策略，后者可按用户精细调整审计。

##### audit_control 文件

审计子系统的若干默认值在 audit_control 中指定：

```
dir:/var/audit
dist:off
flags:lo,aa
minfree:5
naflags:lo,aa
policy:cnt,argv
filesz:2M
expire-after:10M
```

`dir` 条目用于设定一个或多个存储审计日志的目录。若出现多个目录条目，则按顺序填满后依次使用。常见的做法是配置审计，使审计日志存储在专用文件系统上，以防止在文件系统写满时审计子系统与其他子系统之间的干扰。

若将 `dist` 字段设为 `on` 或 `yes`，将在 **/var/audit/dist** 中为所有跟踪文件创建硬链接。

`flags` 字段设定系统范围内有归属事件的默认预选择掩码。在上例中，所有用户的成功和失败登录/注销事件以及认证与授权事件均被审计。

`minfree` 条目定义审计跟踪所在文件系统的最小空闲空间百分比。

`naflags` 条目指定无归属事件应审计的审计类，例如登录/注销过程和认证与授权。

`policy` 条目指定一个以逗号分隔的策略标志列表，用于控制审计行为的各个方面。`cnt` 表示系统在审计失败时应继续运行（强烈推荐使用此标志）。另一标志 `argv` 使 execve(2) 系统调用的命令行参数作为命令执行的一部分被审计。

`filesz` 条目指定审计跟踪在自动终止并轮转跟踪文件之前的最大大小。值为 `0` 则禁用自动日志轮转。若请求的文件大小低于 512k 的最小值，则该值将被忽略并生成一条日志消息。

`expire-after` 字段指定审计日志文件的到期时间，届时将被移除。

##### audit_user 文件

管理员可在 audit_user 中为特定用户指定进一步的审计要求。每行通过两个字段为用户配置审计：`alwaysaudit` 字段指定始终应为该用户审计的事件集，`neveraudit` 字段指定始终不应为该用户审计的事件集。

以下示例条目为 `root` 审计登录/注销事件和成功的命令执行，为 `www` 审计文件创建和成功的命令执行。若与默认的 audit_control 配合使用，`root` 的 `lo` 条目是冗余的，且 `www` 的登录/注销事件也将被审计。

```
root:lo,+ex:no
www:fc,+ex:no
```

### 处理审计跟踪

由于审计跟踪以 BSM 二进制格式存储，FreeBSD 提供了若干内置工具来修改这些跟踪或将其转换为文本。要将跟踪文件转换为简易文本格式，可使用 `praudit`。要精简审计跟踪文件以供分析、归档或打印，可使用 `auditreduce`。此工具支持多种选择参数，包括事件类型、事件类、用户、事件日期或时间，以及被操作的文件路径或对象。

例如，以纯文本形式转储指定审计日志的全部内容：

```
# praudit /var/audit/AUDITFILE
```

其中 *AUDITFILE* 为要转储的审计日志。

审计跟踪由一系列审计记录组成，每条记录由若干令牌（token）构成，`praudit` 按顺序每行打印一个。每个令牌均属特定类型，例如 `header`（审计记录头）或 `path`（名称查找产生的文件路径）。以下为一条 `execve` 事件的示例：

```
header,133,10,execve(2),0,Mon Sep 25 15:58:03 2006, + 384 msec
exec arg,finger,doug
path,/usr/bin/finger
attribute,555,root,wheel,90,24918,104944
subject,robert,root,wheel,root,wheel,38439,38032,42086,128.232.9.100
return,success,0
trailer,133
```

此审计表示一次成功的 `execve` 调用，执行了 `finger doug` 命令。`exec arg` 令牌包含 shell 传递给内核的已处理命令行。`path` 令牌持有内核查找到的可执行文件路径。`attribute` 令牌描述该二进制文件并包括文件模式。`subject` 令牌存储审计用户 ID、有效用户 ID 和组 ID、真实用户 ID 和组 ID、进程 ID、会话 ID、端口 ID 以及登录地址。注意，审计用户 ID 和真实用户 ID 不同，因为用户 `robert` 在执行此命令前切换到了 `root` 账户，但命令仍使用原始认证用户进行审计。`return` 令牌指示执行成功，`trailer` 结束该记录。

也支持 XML 输出格式，可在命令中加入 `-x` 来选用。

由于审计日志可能非常大，可使用 `auditreduce` 选取记录子集。以下示例选取 AUDITFILE 中为用户 `trhodes` 产生的所有审计记录：

```
# auditreduce -u trhodes /var/audit/AUDITFILE | praudit
```

`audit` 组成员有权读取 **/var/audit** 中的审计跟踪。默认情况下该组为空，因此仅有 `root` 用户可以读取审计跟踪。可将用户加入 `audit` 组以委派审计审阅权限。由于追踪审计日志内容的能力提供了关于用户与进程行为的重要洞察，建议谨慎行使审计审阅权限的委派。

#### 使用审计管道进行实时监控

审计管道是可克隆的伪设备，允许应用程序接入实时审计记录流。这主要面向入侵检测和系统监控应用程序的开发者。然而，审计管道设备也为管理员提供了一种便捷方式，使其能够实现实时监控，而不会遇到审计跟踪文件所有权或日志轮转中断事件流的问题。要追踪实时审计事件流：

```
# praudit /dev/auditpipe
```

默认情况下，审计管道设备节点仅可由 `root` 用户访问。要使 `audit` 组成员也能访问，可在 **/etc/devfs.rules** 中添加一条 `devfs` 规则：

```
add path 'auditpipe*' mode 0440 group audit
```

有关配置 devfs 文件系统的更多信息，请参阅 devfs.rules(5)。

审计事件反馈循环容易产生，即每查看一条审计事件就会生成更多审计事件。例如，若审计了所有网络 I/O，并通过 SSH 会话运行 `praudit`，则会以高速率持续产生审计事件流，因为打印的每个事件都将生成另一个事件。因此，建议从没有细粒度 I/O 审计的会话中在审计管道设备上运行 `praudit`。


#### 轮转与压缩审计跟踪文件

审计跟踪由内核写入，由审计守护进程 auditd 管理。管理员不应尝试使用 newsyslog.conf(5) 或其他工具直接轮转审计日志。应改用 `audit` 关闭审计、重配置审计系统并进行日志轮转。以下命令使审计守护进程创建新的审计日志并通知内核切换到新日志。旧日志将被终止并重命名，随后可由管理员操作：

```
# audit -n
```

若 auditd 当前未运行，此命令将失败并产生一条错误消息。

在 /etc/crontab 中加入以下行将每 12 小时计划执行此轮转：

```
0     */12       *       *       *       root    /usr/sbin/audit -n
```

保存 **/etc/crontab** 后此更改即生效。

基于文件大小自动轮转审计跟踪文件可通过 audit_control 中的 `filesz` 实现。

由于审计跟踪文件可能变得非常大，在审计守护进程关闭跟踪后，通常需要压缩或以其他方式归档这些跟踪。audit_warn 脚本可用于针对各种审计相关事件执行自定义操作，包括跟踪在轮转时被正常终止的情况。例如，可在 /etc/security/audit_warn 中添加以下内容，以便在关闭时压缩审计跟踪：

```
#
# Compress audit trail files on close.
#
if [ "$1" = closefile ]; then
        gzip -9 $2
fi
```

其他归档操作可能包括将跟踪文件复制到中央服务器、删除旧跟踪文件，或精简审计跟踪以移除不需要的记录。此脚本仅在审计跟踪文件被正常终止时运行，不会在非正常关机后遗留的未终止跟踪上运行。

## 课后习题

1. 启用进程记账功能（`accton`）后，在系统中执行一系列有规律的操作（创建文件、运行命令、切换用户），然后使用 `lastcomm` 导出记录并逐条解释各字段含义，找出与你操作对应的记录条目。

2. 使用 `sa -u` 生成按用户统计的命令使用报告，分析报告中“cpu”和“re”两列的含义差异，并说明为什么这两个值通常不相等。

3. 讨论进程记账可能带来的隐私问题，以及在多用户系统中部署审计功能时应遵循的伦理原则和法律法规要求。
