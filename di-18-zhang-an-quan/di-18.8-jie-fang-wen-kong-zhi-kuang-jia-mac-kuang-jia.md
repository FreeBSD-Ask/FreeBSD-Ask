# 18.8 强制访问控制框架（MAC 框架）

## 概述

FreeBSD 的强制访问控制（Mandatory Access Control，MAC）框架允许通过可加载的安全策略模块对系统安全进行精细控制。MAC 允许加载访问控制模块以实施安全策略。某些模块为系统的有限子集提供保护，从而强化特定服务。其他模块则跨所有主体与客体提供全面的标签化安全。定义中的“强制”（Mandatory）一词表示控制的执行由管理员和操作系统完成。这与默认的自主访问控制（Discretionary Access Control，DAC）安全机制形成对比，在 DAC 中，执行由用户自行决定。

MAC 策略 **不能覆盖** 传统 Unix 安全机制（文件权限和超级用户检查），只能在 DAC 基础之上施加额外的限制。即使用 MAC 配置了允许访问，如果 DAC 拒绝访问，操作仍然会失败。MAC 策略可以限制 `root` 用户的权限，这是其与 DAC 的关键区别之一。

MAC 框架于 FreeBSD 5.0 中首次引入，是 TrustedBSD 项目的一部分。该框架由 Robert Watson 设计，由 Network Associates Laboratories（NAI Labs）的安全研究部门实现，属于 DARPA/SPAWAR 合同 N66001-01-C-8035（"CBOSS"）的一部分，该合同隶属于 DARPA CHATS 研究计划。

MAC 框架的设计参考了 IEEE POSIX.1e 草案 17（曾被称为 POSIX.1e），该草案旨在为操作系统定义安全扩展，包括访问控制列表（ACL）、审计和强制访问控制。尽管 POSIX.1e 最终未成为正式标准，FreeBSD 的 MAC 实现是 TrustedBSD 项目根据实际需求独立演进的结果，不完全等同于任何已批准的国际或行业标准。

MAC 配置不当可能导致无法访问系统、对用户造成困扰，或无法使用启用 Xorg 。更重要的是，不应仅依赖 MAC 来保护系统。MAC 框架仅增强现有安全策略。如果没有健全的安全实践和定期的安全检查，系统永远无法实现完全安全。

本章中包含的示例仅用于演示目的，**不应** 在生产系统上实施这些示例设置  。实施任何安全策略都需要充分的理解、适当的设计和全面的测试。

虽然本节涵盖与 MAC 框架相关的广泛安全问题，但不会涉及新 MAC 安全策略模块的开发。MAC 框架中包含的许多安全策略模块具有特定特征，这些特征同时用于测试和新模块开发。

## 标准合规

FreeBSD 的 MAC 实现与以下标准及草案相关：

- **IEEE POSIX.1e 草案 17**：mac(3) 用户空间 API 以及 maclabel(7) 标签格式均基于此草案。mac(3) 手册页明确指出，API 与 POSIX 草案的相似性是松散的，因为 POSIX API 无法表达灵活且可扩展的访问控制所需的某些概念。
- **文件系统访问控制列表**：POSIX.1e 草案中定义的 ACL 机制由 FreeBSD 通过 getfacl(1)/setfacl(1) 命令独立实现，不属于 MAC 框架。
- **标签语义**：三种标签化策略（Biba、MLS、LOMAC）的语义分别遵循各自的安全模型理论，这些模型来源于经典的军事安全策略和商用完整性策略研究。

## 关键术语

在涉及 MAC 框架时，使用以下关键术语：

- **区间（compartment）**：一组需要划分或隔离的程序和数据，其中用户被明确授予对系统特定组件的访问权限。区间代表一种分组，例如工作组、部门、项目或主题。区间使得实施按需知密的安全策略成为可能。
- **完整性（integrity）**：可赋予数据的信任级别。随着数据完整性的提高，信任该数据的能力也随之提高。
- **级别（level）**：安全属性的增高或降低设置。级别提高意味着其安全性的提升。
- **标签（label）**：一种可应用于文件、目录或系统中其他项目的安全属性。可将其视为机密性印章。当标签被置于文件上时，它描述了该文件的安全属性，并且仅允许具有相似安全设置的文件、用户和资源进行访问。标签值的含义和解释取决于策略配置。某些策略将标签视为代表客体的完整性或机密性，而其他策略可能使用标签来持有访问规则。
- **多标签（multilabel）**：此属性是文件系统选项，可在单用户模式下使用 tunefs(8) 设置，在引导过程中使用 fstab(5) 设置，或在创建新文件系统时设置。此选项允许管理员对不同客体应用不同的 MAC 标签。此选项仅适用于支持标签化的安全策略模块。
- **单标签（single label）**：一种策略，其中整个文件系统使用标签来对数据流实施访问控制。当未设置 `multilabel` 时，所有文件都将遵循相同的标签设置。
- **客体（object）**：在 **主体** 的指导下信息流经的实体。包括目录、文件、字段、屏幕、键盘、内存、磁存储、打印机或任何其他数据存储或传输设备。客体是数据容器或系统资源。对客体的访问实际上意味着对其数据的访问。
- **主体（subject）**：导致信息在 **客体** 之间流动的任何活动实体，例如用户、用户进程或系统进程。在 FreeBSD 中，这几乎总是一个线程，在某个进程中代表用户执行操作。
- **策略（policy）**：定义如何实现目标的一组规则。策略通常记录某些项目应如何处理。本章将策略视为控制数据和信息流并定义谁有权访问该数据和信息的一组规则。
- **高水位线（high-watermark）**：此类策略允许提高安全级别以访问更高级别的信息。在大多数情况下，进程完成后原始级别会恢复。目前，FreeBSD MAC 框架不包含此类策略。
- **低水位线（low-watermark）**：此类策略允许降低安全级别以访问安全性较低的信息。在大多数情况下，进程完成后用户的原始安全级别会恢复。FreeBSD 中唯一使用此策略的安全策略模块是 mac_lomac(4)。
- **敏感性（sensitivity）**：通常在讨论多级安全（Multilevel Security，MLS）时使用。敏感性级别描述数据的重要程度或保密程度。随着敏感性级别的提高，数据的保密性或机密性的重要程度也随之提高。

## 理解 MAC 标签

MAC 标签是一种可应用于整个系统中主体和客体的安全属性。在设置标签时，管理员必须理解其含义，以防止系统出现意外或非期望行为。客体上可用的属性取决于已加载的策略模块，因为策略模块以不同方式解释其属性。

客体上的安全标签被策略用作安全访问控制决策的一部分。对于某些策略，标签包含做出决策所需的所有信息。在其他策略中，标签可能作为更大规则集的一部分进行处理。

存在两种类型的标签策略：单标签和多标签。默认情况下，系统将使用单标签。管理员应了解每种策略的优缺点，以便实施满足系统安全模型要求的策略。

单标签安全策略仅允许对每个主体或客体使用一个标签。由于单标签策略在整个系统中强制执行一组访问权限，因此它降低了管理开销，但也降低了支持标签化的策略的灵活性。然而，在许多环境中，单标签策略可能已经足够。

单标签策略在某种程度上类似于 DAC，因为 `root` 配置策略使得用户被置于适当的类别和访问级别。一个显著区别是，许多策略模块也可以限制 `root`。对客体的基本控制将下放给组，但 `root` 可以随时撤销或修改设置。

在适当的情况下，可以通过向 tunefs(8) 传递 `multilabel` 在 UFS 文件系统上设置多标签策略。多标签策略允许每个主体或客体拥有自己独立的 MAC 标签。使用多标签还是单标签策略的决定仅对实现标签化功能的策略是必需的，例如 `biba`、`lomac` 和 `mls`。某些策略，如 `seeotheruids`、`portacl` 和 `partition`，完全不使用标签。多标签（`multilabel`）功能仅在 UFS2 文件系统上支持，需要通过在单用户模式下运行 `tunefs -l enable` 来启用。ZFS 等其他文件系统不支持 MAC 多标签。

在分区上使用多标签策略并建立多标签安全模型可能增加管理开销，因为该文件系统中的一切都有一个标签。这包括目录、文件，甚至设备节点。

以下命令将在指定的 UFS 文件系统上设置多标签 `multilabel`。此命令应在单用户模式下执行，且对交换文件系统不是必需的：

```sh
# 必须在单用户模式下执行下条命令
# tunefs -l enable /
tunefs: multilabel remains unchanged as enabled
```

>**注意**
>
>如果未在单用户模式下执行上述命令，将始终显示错误信息 “tunefs: / is not clean - run fsck.”

某些用户在根分区上设置 `multilabel` 标志时曾遇到问题（如标签配置不生效、系统无法正常启动等），具体故障排除步骤见本节末尾“MAC 框架故障排除”。

由于多标签策略是基于每个文件系统设置的，如果文件系统布局设计得当，可能不需要多标签策略。假如要为 FreeBSD Web 服务器设计安全 MAC 模型：此计算机对默认文件系统中的所有内容使用单标签 `biba/high`。如果 Web 服务器需要以 `biba/low` 运行以防止向上写能力，可以将其安装到设置为 `biba/low` 的单独 UFS 的文件系统 **/usr/local**。

### 配置标签

几乎所有方面都能使用基本系统工具来完成标签策略模块配置。这些命令为客体或主体配置，或对配置的控制和验证提供了简单接口。

所有配置可以使用 `setfmac`（用于设置系统客体上的 MAC 标签）和 `setpmac`（用于设置系统主体上的标签）来完成。例如，将 test 的 MAC 标签 `biba` 设置为 `high`：

```sh
# setfmac biba/high test
```

命令执行成功后，将直接返回提示符，无任何错误信息。

常见的错误为 `Permission denied`，通常发生在受限客体上设置或修改标签时。

其他情况可能产生不同的失败。例如，文件可能不属于尝试重新标记客体的用户，客体可能不存在，或者客体可能是只读的。强制策略将不允许进程重新标记文件，原因可能是文件的某个属性、进程的某个属性或提议的新标签值的某个属性。例如，如果以低完整性运行的用户试图更改高完整性文件的标签，或以低完整性运行的用户试图将低完整性文件的标签更改为高完整性标签，这些操作将失败。

系统管理员可以通过使用 `setpmac` 为被调用的进程分配不同的标签来覆盖策略模块的设置：

```sh
# setfmac biba/high test
Permission denied
# setpmac biba/low setfmac biba/high test
# getfmac test
test: biba/high
```

对于当前正在运行的进程（如 sendmail），通常改用 `getpmac`。此命令以进程 ID（PID）代替命令名称。如果用户试图控制不在其访问范围内的文件，受限于已加载策略模块的规则，将显示 `Operation not permitted` 错误。

### 预定义标签

一些支持标签化功能的 FreeBSD 策略模块提供三种预定义标签：`low`、`equal` 和 `high`，其中：

- `low` 被视为客体或主体可拥有的最低标签设置。在客体或主体上设置此标签将阻止其访问标记为 high 的客体或主体。
- `equal` 将主体或客体设置为禁用或不受影响，且仅应放置在被视为豁免于策略的客体上。
- `high` 授予客体或主体在 Biba 和 MLS 策略模块中可用的最高设置。

此类策略模块包括 mac_biba(4)、mac_mls(4) 和 mac_lomac(4)。每个预定义标签确立了不同的信息流指令。请参阅模块的手册页以确定通用标签配置的特性。

### 数值标签

Biba 和 MLS 策略模块支持数值标签，可设置以指示层次控制的精确级别。此数值级别用于将信息划分或分类到不同的分类组中，仅允许访问该组或更高的组级别。例如：

```text
biba/10:2+3+6(5:2+3-20:2+3+4+5+6)
```

可解释为 “Biba Policy Label/Grade 10:Compartments 2, 3 and 6: (grade 5 …”（即“Biba 策略标签/等级 10：区间 2、3 和 6：（等级 5 …）”）。

在此示例中，第一个等级将被视为具有有效区间的有效等级，第二个等级是低等级，最后一个是高等级。在大多数配置中，不需要如此细粒度的设置，因为它们被认为是高级配置。

系统客体仅具有当前等级和区间。系统主体反映系统中可用的权限范围，以及用于访问控制的网络接口。

主体和客体对中的等级和区间用于构造一种称为 **支配（dominance）** 的关系，其中主体支配客体、客体支配主体、两者互不支配，或两者互相支配。“两者互相支配”的情况发生在两个标签相等时。由于 Biba 的信息流特性，用户拥有一组可能对应于项目的区间权限，但客体也拥有一组区间。用户可能必须使用 `su` 或 `setpmac` 来缩减其权限，以便访问不受其限制的区间中的客体。

### 用户标签

用户需配置标签，以确保其文件和进程能与系统定义的安全策略正确交互。这在 **/etc/login.conf** 中使用登录类进行配置。每个使用标签的策略模块都将实现用户类设置。

要设置将由 MAC 强制执行的用户类默认标签，请添加 `label` 条目。包含每个策略模块的示例 `label` 条目如下所示。请注意，在实际配置中，管理员永远不会启用每个策略模块。建议在实施任何配置前先完整阅读本节内容。

```ini
default:
	:welcome=/etc/motd:
	:setenv=MAIL=/var/mail/$,BLOCKSIZE=K:
	:path=~/bin:/sbin:/bin:/usr/sbin:/usr/bin:/usr/local/sbin:/usr/local/bin:
	:manpath=/usr/share/man /usr/local/man:
	:nologin=/usr/sbin/nologin:
	:cputime=1h30m:
	:datasize=8M:
	:vmemoryuse=100M:
	:stacksize=2M:
	:memorylocked=4M:
	:memoryuse=8M:
	:filesize=8M:
	:coredumpsize=8M:
	:openfiles=24:
	:maxproc=32:
	:priority=0:
	:requirehome:
	:passwordtime=91d:
	:umask=022:
	:ignoretime@:
	:label=partition/13,mls/5,biba/10(5-15),lomac/10[2]:
```

用户虽然无法修改默认值，但可以在登录后更改其标签，受限于策略的约束。上例告知 Biba 策略，某进程的最低完整性为 `5`，最高为 `15`，且默认有效标签为 `10`。该进程将以 `10` 运行，直到选择更改标签（可能由于用户使用 `setpmac`），这将被 Biba 限制在已配置的范围内。

对 **/etc/login.conf** 文件进行任何更改后，必须使用 `cap_mkdb` 重建登录类能力数据库。

许多站点用户众多，需划分多个不同的用户类，这需要深入规划，管理可能因此变得困难。

### 网络接口标签

可以在网络接口上设置标签以帮助控制跨网络的数据流。使用网络接口标签的策略，其运行方式与策略对客体的作用方式相同。例如，在 Biba 中处于高设置的用户将不被允许访问标签为 `low` 的网络接口。

在网络接口上设置 MAC 标签时，可以向 `ifconfig` 传递 `maclabel`：

```sh
# ifconfig bge0 maclabel biba/equal
```

此示例将在 `bge0` 接口上设置 `biba/equal` 的 MAC 标签。当使用类似 `biba/high(low-high)` 的设置时，应对整个标签加引号以防止返回错误。

每个支持标签化的策略模块都有一个可调参数，可用于在网络接口上禁用 MAC 标签。将标签设置为 `equal` 将产生类似效果。有关这些可调参数的更多信息，请查看 `sysctl` 的输出、策略手册页以及本章其余部分中的信息。

## 规划安全配置

在实施任何 MAC 策略之前，建议进行规划阶段。在规划阶段，管理员应考虑实施要求和目标，例如：

- 如何对目标系统上的可用信息和资源进行分类。
- 限制对哪些信息或资源的访问，以及应施加的访问限制类型。
- 需要哪些 MAC 模块来实现此目标。

在生产系统上使用 MAC 实施之前，应对受信任系统及其配置进行试运行。由于不同环境有不同的需求和需要，建立完整的安全配置文件将减少系统上线后进行更改的需要。

考虑 MAC 框架如何从整体上增强系统安全性。MAC 框架提供的各种安全策略模块可用于保护网络和文件系统，或阻止用户访问特定端口和套接字。也许策略模块的最佳用途是一次加载多个安全策略模块以提供 MLS 环境。这种方法不同于强化策略，强化策略通常仅针对特定用途强化系统的某些元素。MLS 的缺点是增加了管理开销。

然而，与框架能够为特定配置灵活选择策略并保持较低性能开销所带来的长期收益相比，这种管理开销微不足道。减少对不需要策略的支持可以提高系统的整体性能，同时提供选择的灵活性。良好的实施会考虑整体安全需求，并有效实施框架提供的各种安全策略模块。

使用 MAC 的系统保证用户不会被允许随意更改安全属性。所有用户实用程序、程序和脚本都必须在所选安全策略模块提供的访问规则约束内工作，而 MAC 访问规则的控制权在系统管理员手中。

系统管理员有责任仔细选择正确的安全策略模块。对于需要限制网络访问控制的环境，mac_portacl(4)、mac_ifoff(4) 和 mac_biba(4) 策略模块是良好的起点。对于需要文件系统客体严格机密性的环境，可考虑使用 mac_bsdextended(4) 和 mac_mls(4) 策略模块。

策略决策可以基于网络配置做出。如果仅应允许特定用户访问 ssh(1)，则 mac_portacl(4) 策略模块是一个不错的选择。在文件系统方面，对客体的访问可能对某些用户是机密的，而对其他用户则不然。例如，一个大型开发团队可能被拆分为较小的项目组，其中项目 A 的开发人员可能不被允许访问项目 B 开发人员所编写的客体。然而，两个项目可能都需要访问项目 C 开发人员所创建的客体。使用 MAC 框架提供的不同安全策略模块，可以将用户划分到这些组中，然后授予其对相应客体的访问权限。

每个安全策略模块都有处理系统整体安全的独特方式。模块选择应基于周全的安全策略，该策略可能需要修订和重新实施。了解 MAC 框架提供的不同安全策略模块将有助于管理员为其情况选择最佳策略。

实施 MAC 类似于实施防火墙，因为必须小心防止被完全锁定在系统之外。应考虑恢复到先前配置的能力，并且通过远程连接实施 MAC 时应格外谨慎。

## 可用的 MAC 策略

默认的 FreeBSD 内核包含 `options MAC`。源代码路径位于 **sys/security**。

测试模块后，将模块名称添加到 **/boot/loader.conf** 以便在引导时加载。每个模块也为选择编译自定义内核的管理员提供了内核选项。

### MAC See Other UIDs 策略

模块名称：mac_seeotheruids.ko

内核配置行：`options MAC_SEEOTHERUIDS`

引导选项：`mac_seeotheruids_load="YES"`

mac_seeotheruids(4) 模块扩展了 sysctl 可调参数 `security.bsd.see_other_uids` 和 `security.bsd.see_other_gids`。此选项无需在配置前设置任何标签，并可与其他模块透明地协同工作。

加载模块后，可以使用以下 `sysctl` 可调参数来控制其功能：

- `security.mac.seeotheruids.enabled` 启用该模块并实施默认设置，拒绝用户查看其他用户拥有的进程和套接字的能力。
- `security.mac.seeotheruids.suser_privileged` 当设置为非零值时，允许超级用户（`root`）查看所有用户的进程和套接字。
- `security.mac.seeotheruids.specificgid_enabled` 允许指定组被豁免于此策略。要豁免特定组，请使用 `security.mac.seeotheruids.specificgid=XXX sysctl` 可调参数，将 *XXX* 替换为要豁免的数字组 ID。
- `security.mac.seeotheruids.primarygroup_enabled` 用于豁免特定主组免于此策略。使用此可调参数时，不得设置 `security.mac.seeotheruids.specificgid_enabled`。

### MAC BSD Extended 策略

模块名称：mac_bsdextended.ko

内核配置行：`options MAC_BSDEXTENDED`

引导选项：`mac_bsdextended_load="YES"`

mac_bsdextended(4) 模块实施文件系统防火墙。它提供了标准文件系统权限模型的扩展，允许管理员创建类似防火墙的规则集来保护文件系统层次结构中的文件、实用程序和目录。当尝试访问文件系统客体时，规则列表会被遍历，直到找到匹配的规则或到达末尾。此行为可以使用 `security.mac.bsdextended.firstmatch_enabled` 进行更改。与 FreeBSD 中的其他防火墙模块类似，可以创建一个包含访问控制规则的文件，并由系统在引导时使用 rc.conf(5) 变量读取。

规则列表可以使用 ugidfw(8) 输入，其语法类似于 ipfw(8)。可以使用 libugidfw(3) 库中的函数编写更多工具。

在加载 mac_bsdextended(4) 模块后，可以使用以下命令列出当前规则配置：

```sh
# ugidfw list
0 slots, 0 rules
```

默认情况下，没有定义任何规则，所有内容都是完全可访问的。要创建一条规则来阻止用户的所有访问但不影响 `root`：

```sh
# ugidfw add subject not uid root new object not uid root mode n
```

虽然这条规则非常简单，但这是一种极为不当的做法，因为它阻止所有用户发出任何命令。一个更现实的示例阻止 `user1` 对 `user2` 主目录的所有访问（包括目录列表）：

```sh
# ugidfw set 2 subject uid user1 object uid user2 mode n
# ugidfw set 3 subject uid user1 object gid user2 mode n
```

可以使用 `not uid user2` 替代 `user1`，以为所有用户强制执行相同的访问限制。然而，`root` 用户不受这些规则的影响。

使用此模块时应格外谨慎，因为不当使用可能会阻止对文件系统某些部分的访问。

### MAC DDB 策略

模块名称：mac_ddb.ko

内核配置行：`options MAC_DDB`

引导选项：`mac_ddb_load="YES"`

mac_ddb(4) 策略模块限制内核调试器 ddb(4) 命令提示符下可用的命令子集。允许的命令仅限于那些不读写任意内存位置的命令，以防止提取系统机密，同时保留足够的调试功能以诊断内核 panic。例如，`trace` 或 `show registers` 是允许的，但 `show buffer addr` 不允许。

所有使用 `DB_CMD_MEMSAFE` 标志声明的调试器命令均被允许。该策略还提供验证函数，根据用户提供的参数有条件地允许某些额外命令。

加载后，mac_ddb 还确保只能执行 ddb(4) 调试器后端，而不能执行 gdb(4)。

此策略不使用标签。

### MAC DO 策略

模块名称：mac_do.ko

内核配置行：`options MAC_DO`

引导选项：`mac_do_load="YES"`

mac_do(4) 策略模块允许非特权用户根据管理员配置的规则更改进程凭据（用户 ID 和组 ID）。它支持按 jail 独立配置。

该策略通过 `/usr/bin/mdo` 可执行文件生效，有关此程序的详细信息，请参阅 mdo(1)。规则以 `格式from>to` 的字符串形式指定，每条规则由 `from` 部分（匹配当前凭据）和 `to` 部分（目标凭据）组成，多个规则以分号 `;` 分隔。

以下 sysctl 可调参数可用：

- `security.mac.do.enabled` 启用或禁用 mac_do 策略（默认：1）。
- `security.mac.do.rules` 凭据规则列表，每个 jail 独立配置。

mac_do 支持按 jail 配置规则。每个 jail 必须在其 `/usr/bin/mdo` 路径安装 mdo(1)。jail 参数包括 `mac.do`（`enable`/`disable`/`inherit`）和 `mac.do.rules`。

与传统的 setuid 方式不同，mac_do 不依赖 setuid 程序决定凭据变更是否可接受，而是由内核根据管理员配置的规则做出决策，从而避免 setuid 程序漏洞带来的安全风险。

### MAC Interface Silencing 策略

模块名称：mac_ifoff.ko

内核配置行：`options MAC_IFOFF`

引导选项：`mac_ifoff_load="YES"`

mac_ifoff(4) 模块用于动态禁用网络接口，并防止网络接口在系统引导期间被启动。它不使用标签，也不依赖于任何其他 MAC 模块。

此模块的大部分控制通过以下 `sysctl` 可调参数执行：

- `security.mac.ifoff.lo_enabled` 启用或禁用环回接口 lo(4) 上的所有流量。
- `security.mac.ifoff.bpfrecv_enabled` 启用或禁用伯克利包过滤器接口 bpf(4) 上的所有流量。
- `security.mac.ifoff.other_enabled` 启用或禁用所有其他接口上的流量。

mac_ifoff(4) 最常见的用途之一是在引导序列期间不应允许网络流量的环境中进行网络监控。另一个用途是编写脚本，使用诸如 security/aide 的应用程序，在受保护目录中发现新文件或更改的文件时自动阻止网络流量。

### MAC IP Address Access Control List 策略

模块名称：mac_ipacl.ko

内核配置行：`options MAC_IPACL`

引导选项：`mac_ipacl_load="YES"`

mac_ipacl(4) 策略允许主机使用 sysctl(8) 接口限制 VNET(9) jail 设置 IPv4 和 IPv6 地址的能力。默认情况下，如果策略被强制执行，将拒绝 jail 的所有 IP 地址。

以下 sysctl 可调参数可用：

- `security.mac.ipacl.ipv4` 对 IPv4 地址强制执行 mac_ipacl（默认：1）。
- `security.mac.ipacl.ipv6` 对 IPv6 地址强制执行 mac_ipacl（默认：1）。
- `security.mac.ipacl.rules` IP 地址访问控制列表，格式为 `jid,allow,interface,addr_family,IP_addr/prefix[@jid,...]`。其中 `allow` 为 1 表示允许、0 表示拒绝；`interface` 留空表示匹配所有接口；`addr_family` 为 `AF_INET` 或 `AF_INET6`；`prefix` 为 -1 表示单个 IP，非负值表示子网。

当多条规则适用于同一 IP 地址时，列表中较后定义的规则决定最终结果。

### MAC Port Access Control List 策略

模块名称：mac_portacl.ko

内核配置行：`options MAC_PORTACL`

引导选项：`mac_portacl_load="YES"`

mac_portacl(4) 模块用于限制对本地 TCP 和 UDP 端口的绑定，使得允许非 `root` 用户绑定到 1024 以下的指定特权端口成为可能。

加载之后，此模块在所有套接字上启用 MAC 策略。提供以下可调参数：

- `security.mac.portacl.enabled` 完全启用或禁用此策略。
- `security.mac.portacl.port_high` 设置 mac_portacl(4) 保护的最高端口号。
- `security.mac.portacl.suser_exempt`，当设置为非零值时，将 `root` 用户豁免于此策略。
- `security.mac.portacl.autoport_exempt`，当设置为非零值（默认）时，允许应用程序使用端口 0 进行自动端口分配，绕过规则检查。
- `security.mac.portacl.rules` 以文本字符串形式指定策略，格式为 `rule[,rule,…]`，可根据需要包含任意数量的规则，每条规则的格式为 `idtype:id:protocol:port`。`idtype` 应为 `uid` 或 `gid`。`protocol` 参数可以是 `tcp` 或 `udp`。`port` 参数是允许指定用户或组绑定的端口号。用户 ID、组 ID 和端口参数只能使用数值。

在默认情况下，1024 以下的端口只能由以 `root` 身份运行的特权进程使用。要使 mac_portacl(4) 允许非特权进程绑定到 1024 以下的端口，请按如下设置可调参数：

```sh
# sysctl security.mac.portacl.port_high=1023
# sysctl net.inet.ip.portrange.reservedlow=0
# sysctl net.inet.ip.portrange.reservedhigh=0
```

要避免 `root` 用户受此策略影响，请将 `security.mac.portacl.suser_exempt` 设置为非零值。

```sh
# sysctl security.mac.portacl.suser_exempt=1
```

要允许 UID 为 80 的 `www` 用户绑定到端口 80 而永远不需要 `root` 权限：

```sh
# sysctl security.mac.portacl.rules=uid:80:tcp:80
```

下条示例允许 UID 为 1001 的用户绑定到 TCP 端口 110（POP3）和 995（POP3s）：

```sh
# sysctl security.mac.portacl.rules=uid:1001:tcp:110,uid:1001:tcp:995
```

### MAC NTPD 策略

模块名称：mac_ntpd.ko

内核配置行：`options MAC_NTPD`

引导选项：`mac_ntpd_load="YES"`

策略模块 mac_ntpd(4) 允许 ntpd(8) 以非 `root` 用户身份运行。它授予以 UID 123（`ntpd` 用户）运行的进程以下特权：

- `PRIV_ADJTIME` — 调整系统时间
- `PRIV_CLOCK_SETTIME` — 设置系统时钟
- `PRIV_NTP_ADJTIME` — NTP 时间调整
- `PRIV_NETINET_RESERVEDPORT` — 绑定特权端口
- `PRIV_NETINET_REUSEPORT` — 重用端口

当 ntpd(8) 使用 `-u <user>[:group]` 启动时，它先以 root 权限完成初始化，然后降权切换到指定用户。此策略确保降权后 ntpd 仍能执行时间同步所需的特权操作。

以下 sysctl 可调参数可用：

- `security.mac.ntpd.enabled` 启用 mac_ntpd 策略（默认：1）。
- `security.mac.ntpd.uid` ntpd 用户的数字 UID（默认：123）。

### MAC Partition 策略

模块名称：mac_partition.ko

内核配置行：`options MAC_PARTITION`

引导选项：`mac_partition_load="YES"`

mac_partition(4) 策略根据进程的 MAC 标签将进程放入特定的“分区”。此策略的大部分配置使用 setpmac(8) 完成。`sysctl` 可调参数可用于此策略：

- `security.mac.partition.enabled` 启用 MAC 进程分区的强制执行。

启用此策略后，用户将仅被允许查看自己的进程以及其分区内的任何其他进程，但不被允许使用超出此分区范围的实用程序。例如，处于 `insecure` 类的用户将不被允许访问 `top` 以及许多其他必须生成进程的命令。

此示例将 `top` 添加到 `insecure` 类用户的标签集中。由 `insecure` 类用户生成的所有进程将保持在 `partition/13` 标签内。

```sh
# setpmac partition/13 top
```

此命令显示分区标签和进程列表：

```sh
# ps Zax
```

此命令显示另一用户进程的分区标签和该用户当前正在运行的进程：

```sh
# ps -ZU trhodes
```

除非加载了 mac_seeotheruids(4) 策略，否则用户可以看到 `root` 标签中的进程。

### MAC Priority 策略

模块名称：mac_priority.ko

内核配置行：`options MAC_PRIORITY`

引导选项：`mac_priority_load="YES"`

mac_priority(4) 策略模块根据 group(5) 组成员身份授予调度优先级特权。属于 `realtime` 组（GID 47）的用户或进程被允许以实时调度优先级运行线程和进程。属于 `idletime` 组（GID 48）的用户或进程被允许以空闲调度优先级运行线程和进程。

realtime 策略授予以下内核特权：

- `PRIV_SCHED_RTPRIO` — 实时优先级
- `PRIV_SCHED_SETPOLICY` — 设置调度策略

idletime 策略授予的内核特权为：

- `PRIV_SCHED_IDPRIO` — 空闲优先级

启用 realtime 策略后，特权用户可使用 rtprio(1) 以实时优先级启动进程。启用 idletime 策略后，特权用户可使用 idprio(1) 以空闲优先级启动进程。应用程序也可通过 rtprio(2) 系统调用调整优先级。

以下 sysctl 可调参数可用：

- `security.mac.priority.realtime` 启用 realtime 策略（默认：1）。
- `security.mac.priority.realtime_gid` realtime 组的数字 GID（默认：47）。
- `security.mac.priority.idletime` 启用 idletime 策略（默认：1）。
- `security.mac.priority.idletime_gid` idletime 组的数字 GID（默认：48）。

### MAC 多级安全模块

模块名称：mac_mls.ko

内核配置行：`options MAC_MLS`

引导选项：`mac_mls_load="YES"`

mac_mls(4) 策略通过执行严格的信息流策略来控制系统中主体与客体之间的访问。

在 MLS 环境中，“许可（clearance）”级别连同区间一起设置在每个主体或客体的标签中。由于这些许可级别可以达到大于数千的数字，完整配置每个主体或客体将是一项艰巨的任务。为减轻此管理开销，此策略中包含三种标签：`mls/low`、`mls/equal` 和 `mls/high`，其中：

- 任何标记为 `mls/low` 的内容将具有低许可级别，且不被允许访问更高级别的信息。此标签还防止更高级别的客体向更低级别写入或传递信息。
- `mls/equal` 应放置在应豁免于策略的客体上。
- `mls/high` 是可能的最高许可级别。分配此标签的客体将拥有对系统中所有其他客体的支配权；然而，它们不允许信息泄漏到更低级别的客体。

MLS 提供：

- 分层安全级别，附带一组非层次类别。
- 固定规则“不向上读，不向下写（no read up, no write down）”。这意味着主体可以对其自身级别或更低级别的客体具有读访问权限，但不能对更高级别的客体具有读访问权限。类似地，主体可以对其自身级别或更高级别的客体具有写访问权限，但不对更低级别的客体具有写访问权限。
- 保密性，防止数据不当泄露。
- 为同时处理多个敏感性级别数据而不在机密和保密之间泄漏信息的系统设计提供基础。

存在以下 `sysctl` 可调参数：

- `security.mac.mls.enabled` 用于启用或禁用 MLS 策略。
- `security.mac.mls.ptys_equal` 在创建期间将所有 pty(4) 设备标记为 `mls/equal`。
- `security.mac.mls.revocation_enabled` 在标签更改为更低等级标签后撤销对客体的访问。
- `security.mac.mls.max_compartments` 设置系统上允许的最大区间级别数。

要控制 MLS 标签，请使用 setfmac(8)。为客体分配标签：

```sh
# setfmac mls/5 test
```

获取文件 test 的 MLS 标签：

```sh
# getfmac test
```

另一种方法是在 **/etc/** 路径中创建一个主策略文件，指定 MLS 策略信息，并将该文件提供给 `setfmac`。

使用 MLS 策略模块时，管理员计划控制敏感信息的流动。默认的 `block read up block write down` 将所有内容设置为低状态。所有内容都可访问，管理员逐步增强信息的机密性。

除了三种基本标签选项之外，管理员可以按需对用户和组进行分组，以阻止他们之间的信息流。也许使用描述性词语（如 `Confidential`、`Secret` 和 `Top Secret` 的分类）来查看许可级别中的信息会更容易。一些管理员则根据项目级别创建不同的组。不论分类方法如何，在实施限制性策略之前，必须有一个周全的计划。

MLS 策略模块的一些示例情境包括电子商务 Web 服务器、存放关键公司信息的文件服务器以及金融机构环境。

### MAC Biba 模块

模块名称：mac_biba.ko

内核配置行：`options MAC_BIBA`

引导选项：`mac_biba_load="YES"`

mac_biba(4) 模块加载 MAC Biba 策略。此策略类似于 MLS 策略，不同之处在于信息流规则恰好相反。这是为了防止敏感信息的向下流动，而 MLS 策略防止敏感信息的向上流动。

在 Biba 环境中，“完整性”标签设置在每个主体或客体上。这些标签由层次等级和非层次组件组成。等级升高，其完整性也随之升高。

支持的标签为 `biba/low`、`biba/equal` 和 `biba/high`，其中：

- `biba/low` 被视为客体或主体可拥有的最低完整性。在客体或主体上设置此标签将阻止其对标记为 `biba/high` 的客体或主体的写访问，但不会阻止读访问。
- `biba/equal` 应仅放置在视为豁免于策略的客体上。
- `biba/high` 允许向设置为更低标签的客体写入，但不允许读取该客体。建议将此标签放置在影响整个系统完整性的客体上。

Biba 提供：

- 分层完整性级别，附带一组非层次完整性类别。
- 固定规则“不向上写，不向下读（no write up, no read down）”，与 MLS 相反。主体可以对其自身级别或更低级别的客体具有写访问权限，但不能对更高级别的客体具有写访问权限。类似地，主体可以对其自身级别或更高级别的客体具有读访问权限，但不能对更低级别的客体具有读访问权限。
- 通过防止数据的不当修改来保证完整性。
- 完整性级别而非 MLS 的敏感性级别。

以下可调参数可用于控制 Biba 策略：

- `security.mac.biba.enabled` 用于在目标计算机上启用或禁用 Biba 策略的强制执行。
- `security.mac.biba.ptys_equal` 用于在 pty(4) 设备上禁用 Biba 策略。
- `security.mac.biba.revocation_enabled` 强制在标签更改为支配主体的标签后撤销对客体的访问。

要访问系统客体上的 Biba 策略设置，请使用 `setfmac` 和 `getfmac`：

```sh
# setfmac biba/low test
# getfmac test
test: biba/low
```

完整性（不同于敏感性）用于保证信息不被不可信方控制。这包括在主体与客体之间传递的信息。它确保用户只能修改或访问明确授权其访问的信息。mac_biba(4) 安全策略模块允许管理员配置用户可以查看和调用哪些文件和程序，同时确保这些程序和文件对于该用户在系统上是可信的。

在初始规划阶段，管理员必须准备将用户划分到等级、级别和区域中。一旦启用此策略模块，系统将默认设置为高标签，由管理员为不同用户配置不同的等级和级别。一个好的规划方法可以包括主题，而不是使用许可级别。例如，仅允许开发人员对源代码仓库、源代码编译器和其他开发实用程序拥有修改访问权限。其他用户则被归入其他类别，如测试人员、设计人员或最终用户，仅被允许读访问。

较低完整性的主体无法向较高完整性的主体写入，而较高完整性的主体无法列出或读取较低完整性的客体。将标签设置为尽可能低的等级可能使其对主体不可访问。此安全策略模块的一些适用环境包括受限的 Web 服务器、开发与测试计算机以及源代码仓库。较不适用的实现包括个人工作站、用作路由器的计算机或网络防火墙。

### MAC Low-watermark 模块

模块名称：mac_lomac.ko

内核配置行：`options MAC_LOMAC`

引导选项：`mac_lomac_load="YES"`

与 MAC Biba 策略不同，策略 mac_lomac(4) 允许访问较低完整性客体，但前提是仅在不破坏任何完整性规则的情况下降低完整性级别之后。

Low-watermark 完整性策略的工作方式几乎与 Biba 相同，但使用浮动标签通过辅助等级区间来支持主体降级。此辅助区间采用 `[auxgrade]` 形式。分配带有辅助等级的策略时，使用语法 `lomac/10[2]`，其中 `2` 是辅助等级。

此策略依赖于对所有系统客体的普遍标记（使用完整性标签），允许主体从低完整性客体读取，然后降低主体的标签以防止将来使用 `[auxgrade]` 对高完整性客体进行写入。该策略可能提供比 Biba 更大的兼容性，并且需要更少的初始配置。

与策略 Biba 和 MLS 一样，`setfmac` 和 `setpmac` 用于在系统客体上放置标签：

```sh
# setfmac /usr/home/trhodes lomac/high[low]
# getfmac /usr/home/trhodes lomac/high[low]
```

辅助等级 `low` 是仅由 LOMAC 策略提供的功能。

## 用户锁定

此示例设想了一个相对较小的存储系统，用户数不足五十人。用户将拥有登录能力，并被允许存储数据和访问资源。

对于此场景，策略模块 mac_bsdextended(4) 和 mac_seeotheruids(4) 可以共存，在隐藏用户进程的同时阻止对系统客体的访问。

首先将以下行添加到 **/boot/loader.conf**：

```ini
mac_seeotheruids_load="YES"
```

可以通过在 **/etc/rc.conf** 中添加以下行来激活安全策略模块 mac_bsdextended(4)：

```ini
ugidfw_enable="YES"
```

将在系统初始化时加载 **/etc/rc.bsdextended** 中的默认规则。然而，默认条目可能需要修改。由于此计算机预期仅为用户服务，除了最后两行外，所有内容可以保持注释，以在默认情况下强制加载用户拥有的系统客体。

将所需用户添加到该计算机并重启。为测试目的，请尝试跨两个控制台以不同用户登录。运行 `ps aux` 查看其他用户的进程是否可见。验证对其他用户主目录运行 ls(1) 是否失败。

除非特定 `sysctl` 已被修改以阻止超级用户访问，否则不要尝试使用 `root` 用户进行测试。

添加新用户时，其 mac_bsdextended(4) 规则不会出现在规则集列表中。要快速更新规则集，请卸载安全策略模块，然后使用 kldunload(8) 和 kldload(8) 重新加载。

## 故障排除

本节讨论常见配置错误及其解决方法。

### `multilabel` 标志未在根（**/**）分区上保持启用

当 `multilabel` 标志未在根（**/**）分区上保持启用时，以下步骤可能解决此临时错误：

1. 编辑 **/etc/fstab** 并将根分区设置为 `ro`（只读）。
2. 重启进入单用户模式。
3. 在 **/** 上运行命令 `tunefs -l enable`。
4. 立即运行 `mount -urw /` 并在 **/etc/fstab** 中将 `ro` 改回 `rw`。
5. 重启系统至正常模式。
6. 仔细检查 `mount` 的输出，确保已在根文件系统上正确设置 `multilabel`。

### 无法启动 Xorg

使用 MAC 建立安全环境后，无法启动 Xorg。这可能是由 MAC 策略 `partition` 或某个 MAC 标签策略中的标签设置错误导致的。要调试，请尝试以下步骤：

1. 检查错误消息。如果用户处于 `insecure` 类，则 `partition` 策略可能是问题所在。尝试将用户类恢复为 `default` 类，并使用 `cap_mkdb` 重建数据库。
2. 仔细检查标签策略是否正确设置了用户、Xorg 和 **/dev** 条目。
3. 若两者均无法解决问题，可将错误消息和环境描述发送到 [FreeBSD general questions mailing list](https://lists.freebsd.org/subscription/freebsd-questions)。

### 切换用户时报错

当用户尝试从 `root` 用户切换到系统中的其他用户时，可能出现错误信息 `_secure_path: unable to stat .login_conf`。此消息通常发生在用户具有比其试图切换的目标用户更高的标签设置时。例如，如果 `joe` 的默认标签为 `biba/low`，而 `root` 的标签为 `biba/high`，则 `root` 无法查看 `joe` 的主目录。无论 `root` 是否使用 `su` 成为 `joe`，都会发生这种情况，因为 Biba 完整性模型不允许 `root` 查看设置在更低完整性级别的客体。

### 无法识别 root 用户

系统不再识别 `root`。发生这种情况时，`whoami` 返回 `0`，而 `su` 返回 `who are you?`。

当标签策略被 sysctl(8) 禁用，或者策略模块已被卸载时，可能发生这种情况。如果策略被禁用，需要重新配置登录能力数据库。仔细检查 **/etc/login.conf** 以确保移除了所有 `label` 选项，并使用 `cap_mkdb` 重建数据库。

当策略限制对 **/etc/master.passwd** 的访问时，也可能发生此情况。这通常是由管理员在不符合系统所用通用策略的标签下更改文件导致的。在这些情况下，用户信息将被系统读取，但由于文件继承了新标签，将阻止访问。使用 sysctl(8) 禁用该策略后，应恢复正常。
