# 18.7 第三方漏洞与安全公告

FreeBSD 通过 pkg audit 查询漏洞数据库（VuXML），为已安装第三方软件包提供 CVE 预警。本节演示审计命令的使用，并解读安全公告（Security Advisory）的标准格式。

## 监控第三方安全问题

pkg 轮询着一个安全问题数据库。该数据库由 FreeBSD 安全团队和 ports 开发者更新和维护。可通过 periodic 配置文件维护 pkg 审计数据库，以编程化方式保持更新。

第三方工具的广泛使用也增加了系统入侵的威胁。要随时审计作为 Ports 一部分的第三方工具，管理员可以选择更新数据库并查看已安装软件包的已知漏洞，执行以下命令：

```sh
% pkg audit -F
```

输出应类似于以下内容：

这将包括受影响的版本（按 FreeBSD port 版本），以及可能包含安全公告的其他网站。

## 安全公告

与许多优质操作系统厂商一样，FreeBSD 项目拥有一个安全团队，负责确定每个 FreeBSD 版本的终止生命周期（EoL）日期，并为尚未达到 EoL 的受支持版本提供安全更新。更多关于 FreeBSD 安全团队和受支持版本的信息可在 [FreeBSD 安全页面](https://www.freebsd.org/security)上找到。

安全团队的任务之一是响应 FreeBSD 操作系统中已报告的安全漏洞。漏洞一旦被确认，安全团队将验证修复漏洞所需的步骤，并用修复更新源代码。然后，它将详细信息作为“安全公告”发布。安全公告发布在 [FreeBSD 网站](https://www.freebsd.org/security/advisories/)上，并邮寄到 [FreeBSD 安全通知邮件列表](https://lists.freebsd.org/subscription/freebsd-security-notifications)、[FreeBSD 安全邮件列表](https://lists.freebsd.org/subscription/freebsd-security)和 [FreeBSD 公告邮件列表](https://lists.freebsd.org/subscription/freebsd-announce)。

## 安全公告的格式

以下是一份 FreeBSD 安全公告的示例，来自 <https://www.freebsd.org/security/advisories/FreeBSD-SA-26:08.rpcsec_gss.asc>，报告的内容是 [CVE-2026-4747 Remote code execution via RPCSEC_GSS packet validation](https://www.cve.org/CVERecord?id=CVE-2026-4747)：

```text

                2026-03-26 01:30:12 UTC (stable/13, 13.5-STABLE)
                2026-03-26 01:34:10 UTC (releng/13.5, 13.5-RELEASE-p11)
CVE Name:       CVE-2026-4747

For general information regarding FreeBSD Security Advisories,
including descriptions of the fields above, security branches, and the
following sections, please visit <URL:https://security.FreeBSD.org/>.

I.   Background

Generic Security Services (GSS) is an API which lets applications establish a
private, authenticated communication channel with a server, such as an NFC
server.

RPCSEC_GSS is a module which enables the use of GSS with Sun RPC (rpc(3))
servers.  It is implemented in the kernel by the kgssapi.ko kernel module, and
used by the NFS server to enable Kerberos-based authentication and encryption
of traffic between the server and clients.  In userspace it is implemented by
the librpcsec_gss library.

II.  Problem Description

Each RPCSEC_GSS data packet is validated by a routine which checks a signature
in the packet.  This routine copies a portion of the packet into a stack buffer,
but fails to ensure that the buffer is sufficiently large, and a malicious
client can trigger a stack overflow.  Notably, this does not require the client
to authenticate itself first.

III. Impact

As kgssapi.ko's RPCSEC_GSS implementation is vulnerable, remote code execution
in the kernel is possible by an authenticated user that is able to send packets
to the kernel's NFS server while kgssapi.ko is loaded into the kernel.

In userspace, applications which have librpcgss_sec loaded and run an RPC server
are vulnerable to remote code execution from any client able to send it packets.
We are not aware of any such applications in the FreeBSD base system.

IV.  Workaround

No workaround is available.  Kernels that do not have kgssapi.ko loaded are not
vulnerable.  In userspace, any daemon linked with librpcgss_sec and running an
RPC server is vulnerable.

V.   Solution

Upgrade your vulnerable system to a supported FreeBSD stable or
release / security branch (releng) dated after the correction date.

Perform one of the following:

1) To update your vulnerable system installed from base system packages:

Systems running a 15.0-RELEASE version of FreeBSD on the amd64 or arm64
platforms, which were installed using base system packages, can be updated
via the pkg(8) utility:

# pkg upgrade -r FreeBSD-base
# shutdown -r +10min "Rebooting for a security update"

2) To update your vulnerable system installed from binary distribution sets:

Systems running a RELEASE version of FreeBSD on the amd64 or arm64 platforms,
or the i386 platform on FreeBSD 13, which were not installed using base
system packages, can be updated via the freebsd-update(8) utility:

# freebsd-update fetch
# freebsd-update install
# shutdown -r +10min "Rebooting for a security update"

3) To update your vulnerable system via a source code patch:

The following patches have been verified to apply to the applicable
FreeBSD release branches.

a) Download the relevant patch from the location below, and verify the
detached PGP signature using your PGP utility.

# fetch https://security.FreeBSD.org/patches/SA-26:08/rpcsec_gss.patch
# fetch https://security.FreeBSD.org/patches/SA-26:08/rpcsec_gss.patch.asc
# gpg --verify rpcsec_gss.patch.asc

b) Apply the patch.  Execute the following commands as root:

# cd /usr/src
# patch < /path/to/patch

c) Recompile your kernel and the operating system as described in
<URL:https://www.FreeBSD.org/handbook/kernelconfig.html> and
<URL:https://www.FreeBSD.org/handbook/makeworld.html> and reboot the
system.
```

每份安全公告使用以下格式：

- 每份安全公告均由安全官（Security Officer）的 PGP 密钥签名。安全官的公钥可在 [OpenPGP Keys](https://docs.freebsd.org/en/books/handbook/pgpkeys/#pgpkeys) 中验证。
- 安全公告的名称始终以 `FreeBSD-SA-`（代表 FreeBSD 安全公告）开头，后跟两位数字格式的年份（`23:`），然后是该年份的公告编号（`07.`），再然后是受影响的应用程序或子系统名称（`bhyve`）。
- `Topic` 字段概括了该漏洞。
- `Category` 指的是系统中受影响的部分，可能是 `core`、`contrib` 或 `ports` 之一。`core` 类别意味着该漏洞影响 FreeBSD 操作系统的核心组件。`contrib` 类别意味着该漏洞影响随 FreeBSD 包含的软件，如 BIND。`ports` 类别表示该漏洞影响通过 Ports Collection 提供的软件。
- `Module` 字段指的是组件位置。在此示例中，`bhyve` 模块受到影响；因此，此漏洞影响随操作系统安装的应用程序。
- `Announced` 字段反映了安全公告发布的日期。这意味着安全团队已验证问题存在，并且补丁已提交到 FreeBSD 源代码仓库。
- `Credits` 字段对发现并报告漏洞的个人或组织表示感谢。
- `Affects` 字段说明哪些 FreeBSD 版本受到此漏洞的影响。
- `Corrected` 字段指出版本已被更正的日期、时间、时间偏移量和版本。括号中的部分显示了已合并修复的每个分支，以及来自该分支的相应版本的版本号。版本标识符本身包括版本号，如果适用，还包括补丁级别。补丁级别是字母 `p` 后跟一个数字，表示补丁的序列号，使用户能够跟踪哪些补丁已应用到系统。
- `CVE Name` 字段列出了在公共 [https://www.cve.org/](https://www.cve.org/) 安全漏洞数据库中的公告编号（如果存在）。
- `Background` 字段提供受影响模块的描述。
- `Problem Description` 字段解释该漏洞。这可能包括有关缺陷代码以及该工具如何被恶意使用的信息。
- `Impact` 字段描述了该问题可能对系统产生何种影响。
- `Workaround` 字段指出现有变通办法是否可供无法立即修补系统的系统管理员使用。
- `Solution` 字段提供了修补受影响系统的说明。这是一种经过分步测试和验证的方法，用于使系统得到修补并安全运行。
- `Correction Details` 字段显示每个受影响的 Subversion 或 Git 分支以及包含更正代码的修订版号。
- `References` 字段提供有关漏洞的额外信息来源。

## 课后习题

1. 运行 `pkg audit -F` 获取当前系统中所有存在已知漏洞的软件包列表，从中选取一个高危漏洞，在 CVE 数据库查找其详细信息。

2. 查阅 FreeBSD 官方网站上发布的所有安全公告（FreeBSD-SA），绘制直方图，展示年代和公告数量。

3. 在测试环境中回退一个软件包到存在已知漏洞的旧版本，运行 `pkg audit` 验证漏洞检测结果，然后修复漏洞后再次运行检测，记录完整的流程。
