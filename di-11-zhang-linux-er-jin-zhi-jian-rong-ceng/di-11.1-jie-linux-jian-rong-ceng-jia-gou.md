# 11.1 Linux 兼容层架构

常见的技术认知误区是将 FreeBSD 的 Linux 兼容层等同于虚拟机。

实际测试表明，该兼容层不会引入显著的性能开销，某些场景下部分软件的性能甚至超过原生 Linux 环境。根本原因在于，该兼容层既非指令集模拟器，也非二进制转译层，而是 Linux 应用程序二进制接口（Application Binary Interface，ABI）的系统级实现。

FreeBSD 内核源代码中 Linuxulator 的核心实现位于 [sys/compat/linux/linux_file.c](https://cgit.freebsd.org/src/tree/sys/compat/linux/linux_file.c)。

该模块的功能较为单一：识别 Linux 系统调用，将原始的 Linux 系统调用请求转发至 FreeBSD 系统调用入口点处理。

## 何为 Linuxulator

要理解 FreeBSD 对 Linux 应用程序的兼容机制，需要了解 Linuxulator 这一核心组件。“Linuxulator”的字面含义为“Linux Emulator”，该名称容易与传统指令集模拟器相混淆。然而，Linuxulator 并非传统模拟器，也不是独立的 FreeBSD 用户空间程序。Linuxulator 只是 FreeBSD 官方文档中对某一特定内核模块的非正式称谓，该内核模块的正式标识符为 `linux`。

Linuxulator 与 WSL2 的虚拟机方案和 Wine 的二进制兼容层解释机制存在本质区别。

核心原理如下：FreeBSD 内核能够识别并拦截 Linux 进程的系统调用请求，将其映射到功能等价的 FreeBSD 系统调用，并以 FreeBSD 内核的实现响应这些请求。

> **注意**
>
> **Linuxulator 利用 FreeBSD 内核的系统调用来处理 Linux 进程的系统调用**。

通过 Linuxulator 模块，FreeBSD 内核在系统调用接口上模拟 Linux 内核行为，但实际进程调度和执行仍由 FreeBSD 内核负责，处理系统调用的代码也是 FreeBSD 原生实现。

在进程管理层面，通过 Linuxulator 运行的 Linux 进程在 FreeBSD 内核中按标准 FreeBSD 进程处理，与原生进程无异。

## 什么是 Linux 兼容层

FreeBSD 系统中不存在真实的 Linux 内核，FreeBSD 所声明的 Linux 内核版本号仅具有标识意义，而不具备实际的内核功能约束；该版本号甚至可设置为任意值，例如 `255.255`。

不同的 Linux 软件对内核版本的最低要求不同，其依赖和使用的系统调用接口也存在差异。例如，声明的 Linux 内核版本过低时，Arch Linux 的 chroot 环境可能无法正常初始化，并返回 `kernel too old` 的错误信息。

以下根据 Terry Lambert 用 tlambert@primenet.com 发送给 FreeBSD 聊天邮箱列表的一封电子邮件（邮件 ID：[199906020108.SAA07001@usr09.primenet.com](mailto:199906020108.SAA07001@usr09.primenet.com)）的内容，介绍了 Linux 二进制兼容层的工作原理，原文可能已经佚失。并进行了适当增补更新。

```sh
用户执行命令 (shell)
         |
         v
    +------------+
    | execve(2)  |
    +------------+
         |
         v
  +------------------------+

  +------------------------+
         |
   +-----+-----+
   |           |
   v           v
已知二进制?   否
   |           |

(ELF, a.out,   |
 PE...)         v
   |       shell 执行文件
   v
  ELF 文件?
   |
   v
 brandelf 打 Linux brand
   |
   v
Linux ABI Loader
   |
   v
线程 PCB / syscall context 切换到 Linux 系统调用表
如 sys/amd64/linux/linux_sysent.c
   |
   v
查找依赖二进制 /compat/linux/... -> fallback /...
   |
   v
执行 Linux 二进制程序
```

FreeBSD 拥有抽象层“可执行类加载器（execution class loader）”，在系统调用 `execve(2)` 中嵌入了这一抽象层。

传统上，UNIX 加载器会通过检查幻数（通常是文件的前 4 个或前 8 个字节）来确定文件是否为系统已知的二进制文件。如果是，就调用相应的二进制加载器；如果不是，`execve(2)` 调用将返回错误，shell 会尝试以 shell 命令形式执行它。不论当前使用哪种 shell，默认假定这是一个 shell 命令。

随后，被开发者修改过的 `sh(1)` 会检查文件前两个字符，如果是 `:\n`，则转而调用 shell `csh(1)`。

FreeBSD 拥有一份加载器列表，而非单一加载器，并能降级回退到 `#!` 加载器来执行 shell 解释器或 shell 脚本。

为了提供对 Linux 应用程序二进制接口（ABI）的支持，FreeBSD 将符合 Linux ELF 格式的幻数识别为 ELF 文件，并查找专用 brand（在 ELF 文件的注释部分中），SVR4/Solaris 的 ELF 中没有此 brand。

当 ELF 加载器识别 Linux brand 时，会通过 **线程 PCB 或 syscall context** 切换到 Linux ABI 的系统调用表。所有 Linux 二进制程序的系统调用都通过该表进行索引，内核模块负责处理信号 trampoline、陷阱向量及其他兼容性修补。Linux 系统调用表由内核模块提供，其条目在源码中由 `linux_sysent.c` 生成，映射 Linux 系统调用号到相应的处理函数。

Linux 兼容层还会 **动态搜索根分区**。系统首先在 `/compat/linux/original-path` 下查找文件，如果未找到，则在 `/original-path` 下查找。依赖其它二进制文件的 Linux 程序可以在该机制下运行，例如 Linux 工具链可以在 Linux ABI 支持下执行。即使 Linux 二进制文件不存在相应依赖，也可以回退执行 FreeBSD 二进制文件，并在 `/compat/linux` 目录中提供 `uname(1)` 等工具，使 Linux 程序无法察觉其运行环境非 Linux。

FreeBSD 内核提供了 Linux ABI 的完整系统调用实现。FreeBSD 系统调用表条目和 Linux 系统调用表条目在底层功能上保持一致：文件系统操作、虚拟内存操作、信号传递、System V 进程间通信等。区别在于，FreeBSD 本机二进制程序使用 FreeBSD 的胶水函数，而 Linux 二进制程序使用 Linux 的胶水函数。FreeBSD 胶水函数静态链接到内核，而 Linux 胶水函数可通过内核模块或静态链接访问。

从技术上讲，这并非真正的仿真，而是一种 ABI 实现。有时被称作“Linux 仿真”，只是过去缺少更合适的术语。说 FreeBSD 运行 Linux 二进制程序并不准确，因为这些程序未在 FreeBSD 系统上重新编译。

## 为什么使用 Linux 兼容层并非“苦难哲学”

通过 `kldload linux` 加载模块的做法，不应受到质疑，正如荀子所言：“君子生非异也，善假于物也。”（荀子. 荀子[M]. 北京：中华书局，1954.）

使用 Linux 兼容层同理。类似的技术包括 Linux 上使用 Wine 或 CrossOver，乃至 [ReactOS](https://reactos.org/)；以及 Windows 平台上的 Linux 兼容层和 Android 兼容层，这些方案均已广泛应用。
