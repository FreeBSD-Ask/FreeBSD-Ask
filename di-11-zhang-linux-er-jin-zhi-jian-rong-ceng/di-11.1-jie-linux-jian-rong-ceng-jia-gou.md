# 11.1 Linux 兼容层架构

常见的技术认知误区是将 FreeBSD 的 Linux 兼容层等同于虚拟机。

实际测试表明，该兼容层不会引入显著的性能开销，某些场景下部分软件的性能甚至超过原生 Linux 环境。根本原因在于，该兼容层既非指令集模拟器，也非二进制转译层，而是 Linux 应用程序二进制接口（Application Binary Interface，ABI）的系统级实现。

FreeBSD 内核源代码中 Linuxulator 的核心实现位于 [sys/compat/linux/linux_file.c](https://cgit.freebsd.org/src/tree/sys/compat/linux/linux_file.c)。

该模块的功能较为单一：识别 Linux 系统调用，将原始的 Linux 系统调用请求转发至 FreeBSD 系统调用入口点处理。

## 何为 Linuxulator

要理解 FreeBSD 对 Linux 应用程序的兼容机制，需要了解 Linuxulator 这一核心组件。“Linuxulator”的字面含义为“Linux Emulator”，该名称容易与传统指令集模拟器相混淆。然而，Linuxulator 并非传统模拟器，也不是独立的 FreeBSD 用户空间程序。Linuxulator 只是 FreeBSD 官方文档中对某一特定内核模块的非正式称谓，该内核模块的正式标识符为 `linux`。

Linuxulator 与 WSL2 的虚拟机方案和 Wine 的二进制兼容层解释机制存在本质区别。

核心原理如下：FreeBSD 内核能够识别并拦截 Linux 进程的系统调用请求，将其映射到功能等价的 FreeBSD 系统调用，并以 FreeBSD 内核的实现响应这些请求。

> **注意**
>
> **Linuxulator 利用 FreeBSD 内核的系统调用来处理 Linux 进程的系统调用**。

通过 Linuxulator 模块，FreeBSD 内核在系统调用接口上模拟 Linux 内核行为，但实际进程调度和执行仍由 FreeBSD 内核负责，处理系统调用的代码也是 FreeBSD 原生实现。

在进程管理层面，通过 Linuxulator 运行的 Linux 进程在 FreeBSD 内核中按标准 FreeBSD 进程处理，与原生进程无异。

## 什么是 Linux 兼容层

FreeBSD 系统中不存在真实的 Linux 内核，FreeBSD 所声明的 Linux 内核版本号仅具有标识意义，而不具备实际的内核功能约束；该版本号甚至可设置为任意值，例如 `255.255`。

不同的 Linux 软件对内核版本的最低要求不同，其依赖和使用的系统调用接口也存在差异。例如，声明的 Linux 内核版本过低时，Arch Linux 的 chroot 环境可能无法正常初始化，并返回 `kernel too old` 的错误信息。


FreeBSD 拥有抽象层“可执行类加载器（execution class loader）”，该抽象层直接嵌入在系统调用 execve(2) 的处理逻辑中。

传统上，UNIX 加载器依赖检查幻数（通常为文件首部的 4 或 8 字节）来判断文件是否为系统支持的二进制格式。若匹配，则调用相应的二进制加载器；若不匹配，execve(2) 将返回错误（ENOEXEC）。此时，shell 会接管该文件并默认将其作为 shell 脚本尝试执行。

FreeBSD 并非硬编码单一加载器，而是维护了一个加载器链表。该机制支持按序尝试，并能降级回退到 #!（shebang）加载器来执行指定的 shell 解释器或脚本。

为了提供对 Linux 应用程序二进制接口（ABI）的支持，FreeBSD 的 ELF 加载器在识别出 Linux ELF 格式的幻数后，会进一步在 ELF 的 Note 段中校验专用的 Linux Brand 标记（这是 SVR4/Solaris ELF 所不具备的结构）。

在确认 Linux Brand 后，加载器会在进程的执行上下文中（通过进程结构体的 sysentvec 指针）将默认的系统调用表切换为 Linux ABI 系统调用表。此后，该 Linux 二进制程序发起的所有系统调用均通过此表进行索引。Linux 系统调用表及其对应的信号跳板（signal trampoline）、陷阱向量和兼容性修补逻辑由专门的内核模块提供。其表项映射在源码中由 linux_sysent.c 生成，负责将 Linux 系统调用号路由至对应的内核处理函数。

在文件系统层面，Linux 兼容层实现了一套备用根目录解析机制，其行为表现类似于 union 挂载。在进行路径查找时，系统会优先在 /compat/linux/<original_path> 路径下检索文件；若未命中，则回退至宿主系统的 /<original_path> 下查找。这使得依赖特定共享库的 Linux 动态链接程序能够无缝加载其运行环境。即使缺失部分 Linux 专有依赖，程序也能回退调用 FreeBSD 原生的系统二进制文件，并在 /compat/linux 中通过提供类似 uname(1) 的环境伪装工具，向 Linux 程序提供处于原生 Linux 环境中的错觉。

FreeBSD 内核为 Linux ABI 提供了底层原语支持。FreeBSD 和 Linux 的系统调用在底层功能（如 VFS 操作、虚拟内存操作、信号投递、System V IPC 等）上完全共享同一套内核实现。两者的核心差异仅在于系统调用边界的参数包装与重组机制：FreeBSD 本机程序使用标准的 FreeBSD 系统调用入口，而 Linux 二进制程序则调用 Linux 兼容包装器。前者静态编译于内核中，后者可通过内核模块动态加载或静态链接。

从系统架构的严谨角度界定，这是一种系统调用级别的 ABI 实现，而非指令集层面的“仿真（Emulation）”。早期文献将其称为“Linux 仿真”纯粹是因为当时缺少更准确的分类术语。可以说，由于不需要在 FreeBSD 上进行任何重新编译，CPU 是以原生裸机指令的形式直接执行这些 Linux 二进制程序的，兼容层的作用仅在于完成系统接口的翻译与路由。

## 为什么使用 Linux 兼容层并非“苦难哲学”

通过 `kldload linux` 加载模块的做法，不应受到质疑，正如荀子所言：“君子生非异也，善假于物也。”（荀子. 荀子[M]. 北京：中华书局，1954.）

使用 Linux 兼容层同理。类似的技术包括 Linux 上使用 Wine 或 CrossOver，乃至 [ReactOS](https://reactos.org/)；以及 Windows 平台上的 Linux 兼容层和 Android 兼容层，这些方案均已广泛应用。
